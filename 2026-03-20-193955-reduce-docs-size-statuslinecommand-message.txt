● This is worth thinking through carefully before touching anything. Let me
  break down the problem first.

  Why the files are large:

  The .ispec format bundles three concerns in one file: [human] (intent),
  [policy] (constraints), [spec] (formal contracts — entities, operations,
  queries, events). Every large file carries all three layers.

  The actual usage pattern:

  When coding a specific task you almost never need the whole file. You need:
  - Entity/type definitions (always, small)
  - Either the operation specs (write path) OR the query specs (read path) —
  rarely both
  - Policies only when checking limits/constraints

  ---
  The clearest wins:

  1. Split relationships.ispec into tags.ispec + dependencies.ispec
  These are genuinely separate concerns — different DB tables, different routes,
   different service files. Tag work never requires reading dependency specs and
   vice versa. ~672 lines → ~280 + ~380. This is a real architectural split, not
   artificial.

  2. Split arch.md by when you'd read it:
  - arch-stack.md — ADRs 001-006, 009 (tech choices:
  web/React/Node/PG/Docker/REST). Read when changing the stack or onboarding.
  - arch-features.md — ADRs 007, 008, 010, 011 (domain model, six domains,
  drag-drop, MCP). Read when touching domain design.
  - arch-ops.md — ADRs 012, 013, 014 (backups, env validation, rate limiting).
  Read when touching ops/deployment.
  ~342 lines → ~120 + ~130 + ~90.

  3. Split items.ispec into read path + write path + schema:
  - items-schema.ispec — entities, types, enums, events (~100 lines). Needed by
  everything.
  - items-operations.ispec — the 7 operations (CaptureItem through BlockTask)
  (~350 lines). Read when implementing mutation endpoints.
  - items-queries.ispec — all 8 queries + indexes (~180 lines). Read when
  implementing GET endpoints/dashboard blocks.

  ---
  The tricky parts:

  [human] sections — the intent prose at the top of each ispec is useful
  orientation for any task, but it's only ~30-60 lines per file. It could live
  in a shared items.md alongside the split spec files, or stay duplicated in
  each split file (small enough to tolerate).

  Cross-domain references — relationships.ispec currently imports Item from 
  items. If you split items into schema/operations/queries, the imports in
  relationships need to point at items-schema.ispec. Manageable, but every split
   creates import surface to maintain.

  The doc index — each split creates a new row, but that's the point.
  items-schema.ispec → "entity definitions, use with any items task" is a
